#001
연관 엔티티 고유값을 파생 필드로 저장하는 방식의 문제 (post.writer, comment.writer)
설계 리팩터링 JPA
▼
// 문제

Post.writer, Comment.writer 필드에 Member.email을 직접 복사해 저장하는 방식을 사용했다. 이는 동일한 정보가 두 테이블에 중복 저장되는 데이터 정규화 위반이다. 회원이 이메일을 변경하면 Post, Comment 등 관련된 모든 테이블의 writer 필드를 일괄 업데이트해야 하는 동기화 문제가 생긴다. 

// 문제가 있는 구조
// Member 테이블:  id=1, email=hong@test.com
// Post 테이블:    id=1, writer="hong@test.com", member_id=1
// → 같은 정보가 두 군데 저장됨

@Entity
public class Post {
    private String writer;   // member.email 복사 저장 ← 문제

    @ManyToOne(fetch = LAZY)
    private Member member;   // 같은 정보를 연관관계로도 보유
}
// 해결 방향

파생 필드를 제거하고 연관관계에서 직접 꺼내는 방식으로 변경한다. v1에서는 Object Reference, v2(shop-v2)에서는 ID Reference로 전환하며 완전히 정리한다. 

// v1 — Object Reference에서 직접 접근
post.getMember().getNickname()
post.getMember().getEmail()

// v2 — ID Reference (shop-v2 리팩터링 목표)
public class Post {
    private Long memberId;   // FK만 저장, 객체 참조 없음
    // 필요 시 MemberService.findById(memberId)로 명시적 조회
}
// 배운 점

편의를 위해 파생 필드를 추가하는 것은 단기적으로는 join을 줄여주지만, 데이터 정합성 문제와 동기화 비용이라는 기술 부채를 낳는다.

실무에서도 이러한 이유로 Object Reference(v1) → ID Reference(v2, MSA 대응) 방향으로 설계를 발전시킨다. ID Reference는 Aggregate 경계를 명확히 하고 영속성 컨텍스트 의존도를 낮추는 DDD 원칙과도 일치한다.

#002
Page<T> Redis 캐싱 시 직렬화 문제 — PageImpl + LocalDateTime 역직렬화 실패
성능 설계
▼
// 문제

게시글 목록 조회 결과(Page<PostListDto>)를 @Cacheable로 Redis에 캐싱하려 했으나 두 가지 직렬화 오류가 연속으로 발생했다. 

// 오류 1 — PageImpl 역직렬화 실패
// PageImpl에 기본 생성자가 없어 Jackson이 JSON → 객체 변환 불가
Cannot construct instance of `org.springframework.data.domain.PageImpl`
(no Creators, like default constructor, exist)

// 오류 2 — LocalDateTime 직렬화 실패
// PostListDto.createTime 필드가 LocalDateTime 타입
Java 8 date/time type `java.time.LocalDateTime` not supported by default:
add Module "com.fasterxml.jackson.datatype:jackson-datatype-jsr310"
// 시도한 해결 방향

오류 1 해결을 위해 CachePage<T> 커스텀 DTO를 도입했고, 오류 2 해결을 위해 CacheConfig에 JavaTimeModule을 등록한 커스텀 ObjectMapper를 적용하려 했다. 그러나 우회책이 쌓일수록 코드 복잡도가 급격히 증가했다. 

// 시도 1 — CachePage 커스텀 DTO
public class CachePage<T> implements Serializable {
    private List<T> content;
    private int pageNumber, pageSize, totalPages;
    private long totalElements;
}

// 시도 2 — CacheConfig에 JavaTimeModule 등록
ObjectMapper objectMapper = new ObjectMapper();
objectMapper.registerModule(new JavaTimeModule());
objectMapper.disable(SerializationFeature.WRITE_DATES_AS_TIMESTAMPS);
objectMapper.activateDefaultTyping(..., DefaultTyping.NON_FINAL);
// 최종 결정 — 목록 캐싱 제거

우회책을 계속 쌓는 것보다 목록 캐싱 자체를 포기하는 것이 합리적이라 판단했다. Write-behind 패턴(조회수 캐싱)은 유지하고, 게시글 목록 캐싱은 제거했다. 

// 제거 — @Cacheable, @CacheEvict, CachePage, CacheConfig 복잡도
// 유지 — Write-behind 조회수 캐싱 (StringRedisTemplate + @Scheduled)

@Transactional(readOnly = true)
public Page<PostListDto> searchPosts(PostSearchConditionDto cond, Pageable pageable) {
    return postRepository.searchPosts(cond, pageable); // 단순 DB 조회
}
// 배운 점

Page<T>처럼 Spring 내부 클래스를 Redis에 캐싱하는 것은 생각보다 많은 설정이 필요하다. PageImpl은 기본 생성자가 없고, LocalDateTime은 별도 모듈 등록이 필요하며, 제네릭 타입 정보 보존을 위해 DefaultTyping 설정까지 필요하다.

캐싱 도입 전에 "직렬화 가능한 타입인가"를 먼저 검토해야 한다. 단순 String/Long 값(조회수 카운터)은 캐싱이 쉽지만, 복잡한 객체 그래프는 캐싱 비용이 기대보다 훨씬 클 수 있다.

과설계를 인식하고 단순한 방향으로 롤백하는 결정 자체도 중요한 엔지니어링 판단이다.
